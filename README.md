# team-dev-playbook
# チーム開発プレイブック  
_スクラムと GitHub を使ったハッカソン開発ガイド_

このプレイブックは、「チーム開発（スクラム）と GitHub を活用したアプリケーション開発」ハッカソンに参加するみなさん向けのガイドです。  
ここに書かれていることをすべて完璧に守る必要はありませんが、「困ったときにまずここを見る」チェックリストとして使ってください。

---

## 1. フレームワークや言語は最初に決める

### 起きがちな問題

- チームメンバー各々がそれぞれ画面や機能好きに作りはじめたり、AIの提案をそのまま受け入れると、複数のフレームワークや言語が混ざってしまい、1つの画面として統合できなくなります。

### おすすめの進め方

- **最初のスプリント0または1**で「言語」「フレームワーク」、必要があれば「主要ライブラリ」を決める。
- ライブラリを絶対に使わないといけないというワークではなありません。チームの技術力を大幅に超えた技術選定はアプリケーション開発を遅らせます。
- チーム内の経験に合わせて選ぶ（“かっこよさ”よりも、“全員が触れるかどうか”を優先）。

---

## 2. ブランチの作り方：Issue に紐づける

### よくある失敗

- Issue とブランチが対応しておらず、後から履歴を追えない。
- Issue ページの右側カラムに `create a branch` というリンクがあるので、ここから Issue に紐づく形でブランチを作成する
- ブランチを作成する際に「全角（日本語）」は文字化けするので禁止。また、Issue番号は残したままにする。（例： `1-add-login-page` ）

### おすすめのルール

- 1ブランチでは、できるだけ **1つの目的・1つのIssue** に集中する。

---

## 3. 途中参加メンバーを迷子にしない

> 「途中から参加したんですけど、チームが今何やってるのか分かりません……」

これは、情報がメンバーの頭の中にだけあって、GitHub 上に残っていないときに起こります。

### 情報の置き場所

- **README**
  - プロジェクトの概要（プロダクトゴール含む）  
  - 技術スタック（言語・フレームワークなど）  
  - 起動手順（コマンド）  
- **Wiki やドキュメント用ディレクトリ**
  - 仕様メモ、画面一覧、API 仕様、プロダクトバックログなど  
- **Issue**
  - これからやること・やりたいことを書く場所  
- **Pull Request**
  - 何をどのように変更したかを書く場所  
- **コミットメッセージ**
  - 「この変更は何のため？」が一文でわかるように書く  

### 途中参加者向けの最初の一歩

1. README を読む  
2. `Issues` タブで「Open」のものを眺める  
3. 気になる Issue にコメントして、質問や参加の意思を伝える  

という流れを、チームとしておすすめしておくと親切です。

## 4. GitHub Copilot が使えないとき

### よくある原因

- 個人アカウントで Copilot のプランがになっていない（無料枠や学生特典の有無など）。
- Codespaces に Copilot 拡張機能がインストールされていない。

### 対処の例

1. **GitHub アカウントの Copilot 状態を確認する**
   - GitHub の「Settings」→「Copilot」から、利用できるかどうかを確認する。

2. **Codespaces 上で拡張機能をインストールする**
   - Codespace を開く  
   - 左の拡張機能アイコンから「GitHub Copilot」を検索  
   - インストールし、有効化する  

参考：  
- GitHub Copilot 公式ドキュメント  
  - https://docs.github.com/ja/copilot  
- GitHub Codespaces での Copilot  
  - [https://docs.github.com/ja/enterprise-cloud%40latest/codespaces/reference/using-github-copilot-in-github-codespaces](https://docs.github.com/ja/codespaces/reference/using-github-copilot-in-github-codespaces)

## 5. 時間配分がうまくいかない

### スクラムでの役割のイメージ

- **プロダクトオーナー（PO）**
  - プロダクトバックログ（やりたいことリスト）の中身と優先順位を決める人。
- **スクラムマスター（SM）**
  - チームがスクラムのルールで動けるようにサポートし、障害を取り除く人。
- **開発者（Developers）**
  - 実際に手を動かして機能をつくる人たち。

### おすすめの役割分担

- **時間配分は SM がコントロール**
  - 「設計はあと15分で切り上げよう」  
  - 「次のスプリント計画に進もう」  
  など、イベントのタイムキーパーを担当する。
- **何をやるかは PO が決定**
  - どの Issue を優先するか  
  - スプリントで実装する範囲はどこまでか  
  を PO が決める。

チーム全体が「時間」と「優先順位」を意識して動くのがポイントです。

## 6. AI に頼りすぎて膨大なものを作ろうとしてしまう

### なぜ危ないのか

- AI（GitHub Copilot など）は、とてもリッチで複雑なコードを一気に生成できます。
- しかし、チームの技術力を大きく上回るコードは、
  - バグが出ても直せない  
  - 仕様変更にも対応できない  
  - そもそも何をしているか理解できない  
  という状態になりがちです。

### おすすめの AI の使い方

- **「1プロンプトで全部作らせる」はやめる**
  - 画面、API、DB、認証、エラー処理などを一気に作らせない。
- **1ファンクション・1部位ずつ生成する**
  - 「この関数だけ」「この画面だけ」のように、小さな単位で AI を使う。
  - 生成したらすぐに動作確認し、問題なければコミットする。
- **理解できないコードは採用しない**
  - 「説明できないコード」は、ハッカソンの時間内には扱いきれません。

---

## 7. コミットはセーブ。細かく繰り返してOK

### 細かいコミットのメリット

- どこでバグが入ったか追いやすい。
- 「ちょっと前の動いていた状態」に戻しやすい。
- レビューする人にも、変更の意図が伝わりやすい。

### 具体的なポイント

- **1つのコミットで行う変更は1テーマに絞る**
- 「動いたらコミット」「バグを直したらコミット」くらいの感覚で、こまめに保存してOK。

---

## 8. 計画に時間をかけすぎる

### よくある落とし穴

- 議論し続けて、コードを書く時間が足りなくなる。
- プロダクトバックログを最初から完璧に作ろうとして止まってしまう。

### スクラム的な考え方

- プロダクトゴールやプロダクトバックログのヒントは、**お題スライド**に書かれています。
- スプリント0〜1の時点で、プロダクトバックログが **1〜3項目だけ** でも問題はありません。
- スクラムの利点は、「プロダクトバックログはいつでも手入れ可能」なことです。
  - 作りながら学び、必要に応じて項目や優先順位を更新していきます。

## 9. GitHub Issues にない内容に時間を費やしている

### なぜ問題なのか

- チームから見ると何をしているか分からない。
- ふりかえりのときに、「何に時間を使ったか」が見えず、改善につなげにくい。
- 情報が伝達されずに、同じ内容に再度時間を費やしたり、守られたりしないことがある。

### シンプルなルール

- **「やることはすべて Issue にする」**
  - バグ修正  
  - デザイン検討  
  - ドキュメント整備  
  - リファクタリング  
- Issue を作る → 担当者（Assignee）を設定する → 作業を始める、という流れをチームで統一する。

---

## 10. スコープが大きすぎて終わらない

### よくあるパターン

- 壮大な事業計画や、たくさんの機能を持つアプリを最初から全部作ろうとする。
- 結果として、どの機能も中途半端だったり、そもそも開発ができずにデモで見せられる状態にならない。

### 意識したいこと

- まずは「**スプリント1のゴール**」に集中する。
- 「ユーザーが実際に触れる最小限の機能（MVP）」を決めて、それを最優先で作る。
---

## 11. スプリントゴールがない・弱い

### どうなってしまうか

- 「とりあえず作れそうなところから作る」という進め方になってしまう。
- できあがったものがバラバラで、一貫性のない製品になる。
- デモで使えず、「ユーザーにとっての価値」がほぼゼロになってしまう。

### 良いスプリントゴールの例

- 「ログインして、自分のタスク一覧だけは見られるようにする」
- 「写真を1枚アップロードして、タイトルをつけて保存できるようにする」

スプリントゴールは短く、「デモでこういう体験を見せたい」と説明できるレベルだとベストです。

---

## 12. Issue の書き方が雑

### よくある問題

- タイトルだけで本文がない。
- タイトルを読んでも、何をしたいのか分からない。
- Assignee, Label, Milestone が設定されていない。

### シンプルなテンプレ

Issue の本文は、最低でも次の3つを書くと親切です。

1. **目的（Why）**  
   - なぜこのIssueが必要なのか？
2. **やること（What）**  
   - 箇条書きで、やる作業を列挙。
3. **完了条件（Done）**  
   - どうなっていればこのIssueをクローズできるか？

さらに、ラベルやマイルストーンを使うと、  
「これはバグか？ 機能追加か？」  
「どのスプリントで対応するか？」  
といった情報が一目で分かります。

また、Issueのタイトルは「ダッシュボード」のように単語を登録するだけでは足りません。  
「ダッシュボードに必要な項目を洗い出す」「ダッシュボードをデザインする」「ダッシュボードを実装する」など、詳細をIssueタイトルに設定しましょう。  
また、LabelやMilestoneも常に最新の状態を保ちましょう。

---

## 13. Issue がクローズされない

### なぜ危険なのか

- 実際には終わっているのに、Issue 上は「終わっていない」状態になり、進捗が見えない。
- 逆に、「まだ終わっていないのにクローズされたまま」で、バグが放置されることもある。

### 習慣にしたいこと

- Pull Request がマージされたタイミングで、関連する Issue を自動クローズする。  
  - PR の本文に `Closes #XX` と書く。
- デイリースクラムやふりかえりで、
  - 「このIssueはもう終わってるよね？」  
  - 「これはまだ途中だよね？」  
  を確認する時間を少し取る。
