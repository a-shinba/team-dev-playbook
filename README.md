## チーム開発プレイブック

_スクラムと GitHub を使ったハッカソン開発ガイド_

このプレイブックは、「チーム開発（スクラム）と GitHub を活用したアプリケーション開発」ハッカソンに参加するみなさんのためのガイドです。困ったときに、まず確認するチェックリストとして使ってください。

### 開発の流れ

ハッカソンでは、次の 6 つのフェーズを意識してチーム開発を進めます。順番どおりに進まないこともありますが、今の状況に近いフェーズから確認してみましょう。

| フェーズ | このフェーズで確認すること |
| --- | --- |
| 1. チームビルディング | 役割と情報共有の方法をそろえる |
| 2. 開発準備 | 使う技術と開発環境をそろえる |
| 3. 計画 | プロダクトバックログ（PB）、スプリントゴール、Issue を決める |
| 4. 開発中 | Issue に沿って、小さな単位で開発する |
| 5. レビュー・マージ | 変更を確認し、安全に取り込む |
| 6. ふりかえり | チームの動き方を見直し、次の開発につなげる |

### 1. チームビルディング

最初に、チーム内の役割と情報共有の方法をそろえます。途中から参加する人も含め、全員が同じ情報を見て動ける状態を目指します。

#### 1-1. 役割と時間配分を決める

##### スクラムでの役割のイメージ

- **プロダクトオーナー（PO）**
  - プロダクトバックログ（やりたいことリスト）の内容と優先順位を決める人。
- **スクラムマスター（SM）**
  - チームがスクラムのルールで動けるように支え、作業を妨げる問題を取り除く人。
- **開発者（Developers）**
  - 実際に手を動かして機能をつくる人たち。

##### おすすめの役割分担

- **時間配分は SM が管理する**
  - 「設計はあと 15 分で切り上げよう」
  - 「次のスプリント計画に進もう」
  - イベントごとの時間を管理する。
- **何を作るかは PO が決める**
  - 優先する Issue
  - スプリントで実装する範囲

チーム全員が「残り時間」と「優先順位」を意識して動くことがポイントです。

#### 1-2. 途中参加メンバーを迷子にしない

> 「途中から参加したんですけど、チームが今何やってるのか分かりません……」

これは、チームの情報がメンバーの頭の中にだけあり、GitHub 上に残っていないときに起こります。

##### 情報の置き場所

- **README**
  - プロジェクトの概要（プロダクトゴールを含む）
  - 技術スタック（言語・フレームワークなど）
  - 起動手順（コマンド）
- **Wiki やドキュメント用ディレクトリ**
  - 仕様メモ、画面一覧、API 仕様、プロダクトバックログなど
- **Issue**
  - これからやること・やりたいことを書く場所
- **Pull Request**
  - 何をどのように変更したかを書く場所
- **コミットメッセージ**
  - 「この変更は何のためか」が一文で分かるように書く

##### 途中参加者向けの最初の一歩

1. README を読む
2. `Issues` タブで「Open（未完了）」のものを見る
3. 気になる Issue にコメントして、質問や参加の意思を伝える

この流れをチームのルールとして共有しておくと、途中から参加する人も動きやすくなります。

### 2. 開発準備

開発を始める前に、全員が同じ技術と環境で作業できるように準備します。ここをそろえると、あとからコードをまとめやすくなります。

#### 2-1. フレームワークや言語は最初に決める

##### 起きがちな問題

- チームメンバーがそれぞれ好きな画面や機能を別々に作り始めたり、AI の提案をそのまま採用したりすると、複数の言語やフレームワークが混ざってしまいます。その結果、1 つのアプリとしてまとめられなくなることがあります。

##### おすすめの進め方

- **最初のスプリント（スプリント 0 または 1）**で、使う「言語」と「フレームワーク」を決める。必要であれば、「主要なライブラリ」も決める。
- このワークでは、ライブラリを必ず使う必要はありません。チームの技術力を大きく超える技術を選ぶと、開発が遅れる原因になります。
- 見た目の「かっこよさ」よりも「全員が使えるか」を優先し、チームの経験に合うものを選ぶ。
- 安定して動作確認ができ、開発のスピードも保ちやすいため、このワークショップでは HTML、CSS、JavaScript の構成をおすすめします。
  - ほかの言語やフレームワークを使っても問題ありません。ただし、チーム全員が同じ環境で動作確認や開発をできることが必要です。トラブルも、基本的にはチーム内で解決します。

#### 2-2. GitHub Copilot を使える状態にする

##### よくある原因

- 個人アカウントが Copilot を利用できるプランになっていない（無料枠や学生向け特典の有無など）。
- Codespaces に Copilot 拡張機能がインストールされていない。

##### 対処の例

1. **GitHub アカウントの Copilot 状態を確認する**
   - GitHub の「Settings」→「Copilot」を開き、利用できる状態か確認する。

2. **Codespaces 上で拡張機能をインストールする**
   - Codespace を開く
   - 左の拡張機能アイコンから「GitHub Copilot」を検索
   - インストールして有効にする

参考リンク：

- [GitHub Copilot 公式ドキュメント](https://docs.github.com/ja/copilot)
- [GitHub Codespaces で GitHub Copilot を使う](https://docs.github.com/ja/codespaces/reference/using-github-copilot-in-github-codespaces)

### 3. 計画（プロダクトバックログ・スプリント）

プロダクトバックログ（PB：やりたいことリスト）から、今回のスプリントで作るものを選びます。限られた時間で何を完成させるかを、チームで具体的に決めます。

#### 3-1. 計画に時間をかけすぎない

##### よくある落とし穴

- 話し合いを続けすぎて、コードを書く時間が足りなくなる。
- やることリストである「プロダクトバックログ」を、最初から完璧に作ろうとして止まってしまう。

##### スクラム的な考え方

- プロダクトゴールやプロダクトバックログのヒントは、**お題スライド**に書かれています。
- スプリント 0〜1 の時点では、プロダクトバックログが **1〜3 項目だけ** でも問題ありません。
- スクラムの良いところは、プロダクトバックログをいつでも見直せることです。
  - 作りながら学び、必要に応じて項目や優先順位を変えていきます。

#### 3-2. 作る範囲（スコープ）を小さくする

##### よくあるパターン

「スコープ」とは、作る範囲のことです。

- 最初から、壮大な事業計画や、たくさんの機能を持つアプリをすべて作ろうとする。
- その結果、どの機能も中途半端になり、デモで見せられる状態まで完成しない。

##### 意識したいこと

- まずは「**スプリント 1 のゴール**」に集中する。
- 「ユーザーが実際に使える最小限の機能（MVP）」を決め、それを最優先で作る。

#### 3-3. スプリントゴールを決める

##### ゴールがないとどうなるか

- 「とりあえず作れそうなところから作る」という進め方になってしまう。
- できあがった機能がバラバラになり、まとまりのない製品になる。
- デモで使えず、ユーザーにとって役立つものにならない。

##### 良いスプリントゴールの例

- 「ログインして、自分のタスク一覧だけは見られるようにする」
- 「写真を 1 枚アップロードして、タイトルをつけて保存できるようにする」

スプリントゴールは短くし、「デモでこのような体験を見せたい」と説明できる内容にします。

#### 3-4. 分かりやすい Issue を作る

##### よくある問題

- タイトルだけで本文がない。
- タイトルを読んでも、何をしたいのか分からない。
- 担当者（Assignee）、ラベル（Label）、マイルストーン（Milestone）が設定されていない。

##### シンプルなテンプレ

Issue の本文には、最低でも次の 3 つを書くと分かりやすくなります。

1. **目的（Why）**
   - なぜこの Issue が必要なのか？
2. **やること（What）**
   - やる作業を箇条書きで書く。
3. **完了条件（Done）**
   - どのような状態になれば、この Issue をクローズできるか？

さらに、ラベルやマイルストーンを使うと、「これはバグか？ 機能追加か？」「どのスプリントで対応するか？」といった情報が一目で分かります。

Issue のタイトルは、「ダッシュボード」のように単語だけを書くのでは不十分です。「ダッシュボードに必要な項目を洗い出す」「ダッシュボードをデザインする」「ダッシュボードを実装する」など、具体的な作業が分かるタイトルにしましょう。

ラベルやマイルストーンも、常に最新の状態にしておきましょう。

### 4. 開発中

計画した Issue をもとに、変更を小さく分けて開発します。作業内容を GitHub に残し、チームで進み具合を確認できる状態を保ちます。

#### 4-1. ブランチを Issue に紐づける

##### Issue とブランチを紐づける

- Issue とブランチが対応していないと、あとから変更の履歴を追いにくくなります。
- Issue ページの右側にある `create a branch` をクリックし、その Issue に紐づくブランチを作ります。
- ブランチ名に全角文字（日本語）を使うと文字化けすることがあるため、半角英数字を使います。Issue 番号は消さずに残します（例：`1-add-login-page`）。

##### おすすめのルール

- 1 つのブランチでは、できるだけ **1 つの目的・1 つの Issue** だけに取り組みます。

#### 4-2. やることはすべて Issue に書く

##### Issue に書かないと起きること

- チームから見ると、誰が何をしているのか分からない。
- ふりかえりのときに、「何に時間を使ったか」が見えず、改善につなげにくい。
- 情報が共有されないと、同じ作業を繰り返したり、決めたことが守られなかったりすることがあります。

##### シンプルなルール

- **「やることは、すべて Issue に書く」**
  - バグ修正
  - デザイン検討
  - ドキュメント整備
  - コードの整理（リファクタリング）
- 「Issue を作る → 担当者（Assignee）を決める → 作業を始める」という流れをチームで統一する。

#### 4-3. AI は小さな単位で使う

##### 大きな単位で使うと危ない理由

- AI（GitHub Copilot など）は、高機能で複雑なコードを一気に生成できます。
- しかし、チームが理解できる範囲を大きく超えるコードには、次のような問題があります。
  - バグが出ても直せない。
  - 仕様変更に対応できない。
  - コードが何をしているのか理解できない。

##### おすすめの AI の使い方

- **「1 回の指示（プロンプト）で全部作らせる」はやめる**
  - 画面、API、DB、認証、エラー処理などをまとめて作らせない。
- **1 つの関数・1 つの部分ずつ生成する**
  - 「この関数だけ」「この画面だけ」のように、小さな単位で AI を使う。
  - 生成したコードはすぐに動作確認し、問題がなければコミットする。
- **理解できないコードは使わない**
  - 自分で説明できないコードは、ハッカソンの時間内には扱いきれません。

#### 4-4. コミットは細かく繰り返す

コミットは、コーディングのセーブ機能です。

##### 細かいコミットのメリット

- どこでバグが入ったか追いやすい。
- 「ちょっと前の動いていた状態」に戻しやすい。
- レビューする人にも、変更の目的が伝わりやすい。

##### 具体的なポイント

- **1 つのコミットで行う変更は 1 テーマに絞る**
- 「動いたらコミット」「バグを直したらコミット」という感覚で、こまめに保存する。

### 5. レビュー・マージ

完成した変更をチームメンバーと確認し、問題がなければ取り込みます。Pull Request と Issue の状態も、実際の進み具合に合わせて更新します。

#### 5-1. Pull Request はレビュー後にマージする

- プルリクエストを作成した本人だけで判断し、そのままマージしない。
- 必ずチームメンバーをレビュワーに指定する。レビューを受け、問題がなければマージしてもらう。
- プルリクエストの本文（説明文）にスクリーンショットを添付するなど、変更内容を分かりやすくしてレビュワーの負担を減らす。

#### 5-2. Issue の状態を最新に保つ

##### 状態が更新されないと起きること

- 実際には終わっているのに、Issue 上は「終わっていない」状態になり、進捗が見えない。
- 逆に、まだ終わっていない Issue をクローズすると、バグや作業が放置されることもある。

##### 習慣にしたいこと

- Pull Request がマージされたら、関連する Issue を自動でクローズする。
  - PR の本文に `Closes #XX` と書く。
- デイリースクラムやふりかえりで、各 Issue が完了したか確認する時間を作る。
  - 「この Issue はもう終わっているよね？」
  - 「これはまだ途中だよね？」

### 6. ふりかえり

スプリントの終わりに、できた機能だけでなく、チームの動き方も振り返ります。次のスプリントで続けることと改善することを見つけます。

#### 6-1. チームの動き方をふりかえる

ふりかえり（レトロスペクティブ）では、完成した機能だけでなく、チームがどのように動いたかに注目します。

- **△ 機能だけで良し悪しを決める**
  - 「〇〇機能が実装できたから良かった」
  - 「〇〇機能が実装できなかったから悪かった」
- **○ チームの動き方を振り返る**
  - 「分からないことをすぐに質問できる雰囲気と習慣を作ったので、一人で悩む時間が減り、効率よく作業できた」
  - 「ただし、質問に答える人の作業時間が減るので、質問用のテンプレートがあるとよかったかもしれない」

チームの動き方に注目すると、次の開発にも経験を生かせます。

- 「良かった点」を続けることで、同じ質の開発を再現できる。
- 「改善点」を直すことで、同じ原因の失敗を防げる（スクラム用語では「適応」）。

スクラムでは、このようにチームの動き方を振り返る時間が**必ず**設けられています。大切な時間なので、レトロスペクティブを飛ばさないでください。

「良かった」「悪かった」で終わらず、「次からどのような仕組みにするか」まで考えると、より良い「検査・適応」につながります。

- 例えば、「質問が来ていないか気にする」だけでなく、「スプリント終了の○分前に、SM の通知で必ずチャットを確認する時間を作る」など。
